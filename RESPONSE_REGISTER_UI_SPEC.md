# Response Register UI Specification

## Purpose

Transform the Bellbrook Advocacy Portal from a submission generator into a community memory system.

The Response Register captures what happened after a package was sent.

## Resident Workflow

1. Create Community Impact Record
2. Generate Submission Package
3. Send Package
4. Receive Response
5. Record Response
6. Track Outcome

## Response Register Fields

### Submission Details

- Submission ID
- Date created
- Date sent
- Recipient category
- Recipient name
- Submission method

### Acknowledgement Tracking

- Acknowledgement received (Yes/No)
- Acknowledgement date
- Acknowledgement summary

### Response Tracking

- Response received (Yes/No)
- Response date
- Response summary
- Documents received

### Question Tracking

- Questions answered
- Questions partially answered
- Questions unanswered

### Outcome Tracking

- No response
- Referred elsewhere
- Information provided
- Action promised
- Action completed
- Matter unresolved

### Status

- Created
- Sent
- Awaiting acknowledgement
- Awaiting response
- Responded
- Escalated
- Closed

## Community Memory Principle

Every submission should leave a trace.

The system preserves:

- What was reported
- Who received it
- What response was given
- What remains unresolved

## Future Dashboard Metrics

- Records submitted
- Responses received
- Outstanding responses
- Questions answered
- Questions unanswered
- Escalations created
- Matters resolved

## Transparency Rule

Responses are stored separately from resident testimony.

The system must clearly distinguish:

- Resident testimony
- Portal-generated content
- Recipient responses
- Community summaries

## Long-Term Goal

Create an institutional memory that survives changes in residents, councillors, staff, committees, governments, and advocacy groups.
